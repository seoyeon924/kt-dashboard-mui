1
STEP ONE
@data-ingestion
"이 데이터로 무엇을 할 수 있는가"를
팀 전체가 같은 이해로 시작하게 만듭니다
claude-haiku-4-5
읽는 것 (Input)
  • runs/latest/input/ 의 데이터 파일
  • CSV · TSV · Excel · JSON · PDF 형식 자동 감지
만드는 것 (Output)
  • 컬럼별 타입 · 고유값 수 · 결측치 비율 표
  • 이상치 · 중복 · 형식 이슈 심각도 진단
    높음 20%+ 중간 5~20% 낮음 ~5%
  • 다음 에이전트(@eda)에게 전달할 주의사항
핵심 원칙
분석하지 않는다 — 구조와 품질 파악이 전부다.
문제를 축소하지 않고 있는 그대로 보고한다.
2
STEP TWO
@eda
숫자를 나열하는 것이 아니라
"왜 중요한가"로 연결되는 패턴을 찾습니다
claude-haiku-4-5
읽는 것 (Input)
  • 01_dataset_profile.md (이전 단계 산출물)
  • 원본 CSV 직접 읽기
만드는 것 (Output)
  • 일변량 분석 — 각 컬럼의 분포·평균·이상값
  • 다변량 분석 — 컬럼 간 상관관계·그룹 비교
  • 적합한 분석 방법론 후보 테이블
  • 핵심 발견 3가지 (수치 근거 포함)
  • 이 데이터로 답할 수 있는 분석 질문 3~5개
핵심 원칙
도메인 고정 가정 금지 — 커머스 방법론을 금융·통신 데이터에 강제하지 않는다. 방법론은 데이터가 보여주는 것을 따른다.
3
STEP THREE
@problem
EDA가 발견한 패턴을
"해결해야 할 비즈니스 문제"로 바꿉니다
claude-haiku-4-5
읽는 것 (Input)
  • 02_eda_report.md (이전 단계 산출물)
만드는 것 (Output)
  • 측정 가능한 비즈니스 문제 정의 2~3개
  • 메인 문제의 MECE 원인 계층 구조
    문제 → 예상 원인 → 원인의 원인 → 해결방안 → 목표 지표
  • 핵심 분석 질문 Top 3 (검증 방법 + 의사결정 연결)
핵심 원칙
"무엇이 일어났는가"가 아니라 "왜 일어났는가"를 묻는다.
질문은 반드시 의사결정으로 이어져야 한다.
4
STEP FOUR
@metrics
"매출"이 아니라
"세금 제외 순매출(월 합계, 전월 대비)"처럼
모호함을 없앱니다
claude-haiku-4-5
읽는 것 (Input)
  • 03_problem_definition.md
  • 원본 CSV 컬럼 확인
만드는 것 (Output)
  • KPI 목록 — 결과 지표 · 선행 지표 · 진단 지표 · 파생 지표
  • 각 KPI의 계산식 + 단위 + 현재값 + 비교 기준
  • 액션 가이드: 지표가 X% 나빠지면 → 할 것
핵심 원칙
선행 지표를 반드시 1개 이상 포함한다 — 결과 지표만 있으면 너무 늦게 안다. 절대값과 상대값은 절대 혼용하지 않는다.
5
STEP FIVE
@analysis
숫자를 읽는 것이 아니라
변화의 의미를 해석합니다
claude-haiku-4-5
읽는 것 (Input)
  • 03_problem_definition.md + 04_kpi_summary.md
  • 원본 CSV (필요 시 직접 접근)
만드는 것 (Output) · General → Specific 흐름
  • 전체 현황 → 주요 분해 → 원인 탐색 → 이상 발견 → 기회 발견
  • 핵심 인사이트 3개 (발견 · 근거 · 임팩트 · 한계)
  • 우선순위별 추천 액션 — P1 즉시 / P2 2~4주 / P3 모니터링
  • 대시보드용 한 줄 메시지
핵심 원칙
추측은 추측이라고 밝힌다 — "~로 보인다"와 "~이다"를 구분한다.
인사이트는 행동으로 이어져야 한다.
6
STEP SIX
@dashboard
"예쁜 시각화"가 아니라
"3초 안에 핵심이 읽히는
대시보드"를 만듭니다
claude-sonnet-4-6
읽는 것 (Input)
  • dashboard-sample.html 베이스 템플릿
  • 05_analysis_report.md + 04_kpi_summary.md
만드는 것 (Output)
  • KPI 카드 · 추세 차트 · 비교 차트 · 스토리 블록
  • 탭 구조 (임원용 Overview + 실무자용 세부 탭)
  • 시나리오 시뮬레이터 슬라이더 위젯
  • 순수 SVG/Canvas — CDN 라이브러리 없음
핵심 원칙
차트 제목은 질문형 또는 결론형 —
"매출 현황"이 아니라 "왜 마진이 4배 낮은가?"
데이터가 다르면 대시보드 구조도 달라진다.
7
STEP SEVEN
@report
분석 팀의 모든 작업을
임원이 5분 안에 읽을 수 있는
보고서로 압축합니다
claude-haiku-4-5
읽는 것 (Input)
  • 파이프라인 전체 산출물 5개 파일
  • 대시보드 품질 검토 (임원 · 실무자 · 데이터 신뢰성 3관점)
만드는 것 (Output)
  • Executive Summary — 한 줄 요약 + 핵심 KPI 현황
  • 주요 발견 3개 (수치 + 임팩트)
  • 액션 플랜 테이블 — P1 즉시 / P2 단기 / P3 모니터링
  • 대시보드 품질 검토 체크리스트
핵심 원칙
"무엇을 발견했는가"보다 "무엇을 해야 하는가"가 중심. 임원은 맥락을 모른다 — 자기완결적으로 쓴다.
정리
7개 에이전트가 지키는
공통 원칙
KPI 정의가 먼저
이름 · 계산 기준 · 단위를 확정한 뒤 분석 시작. 정의 없이 숫자를 내지 않는다.
절대값 / 상대값 혼용 금지
"+100건"과 "+5%"는 의미가 다르다. 비교 기준 없는 숫자는 내지 않는다.
차트는 질문에 답한다
예쁨보다 명확함 우선. 카드 제목은 결론형 또는 질문형으로만.
도메인 독립성
커머스용 가정을 KT 통신 데이터에 강제하지 않는다. 데이터가 보여주는 것만.
